Cloud

Cloud computing is the on-demand availability of computer system resources, especially data storage and computing power, without direct active management by the user. The term is generally used to describe data centers available to many users over the Internet.

    Module 2 - Big Data, Open Source Software and the Cloud
    Web Servers and Web Sites
    The story of IoT is one of devices that gather data and connect to servers that aggregate that data into powerful data stores. We trace the rise of the Browser as a data integrator that allows us to scale our systems across the world.
    We show how to gather data from hundreds or thousands of sensors or to control those sensors just using a Web Servers.
    At the same time we trace the effect of Open Source Software as a driver of rapid innovation. In particular we look at Node.js, JavaScript and NPM (Node Package Manager) in allowing companies to build powerful web platforms.
    In our exercises we look the importance of Web Servers, URLs and Routes that allow us to connect to and control devices across the world. Using these we can control a camera in our home or in Hong Kong with the same ease - physical distance being no problem.
    We also look at two standards for communication, namely HTTP and Web Sockets. HTTP is a simple protocol, which is so small it can be written on a business card. It is the standard for communicating with a Browser to a Web Server. Its "header" contains information about the browser, the machine IP and port address that its sent from and the machine IP and port address of its destination. For security HTTPS also includes a mechanism for encryption, which makes communication more secure. The downside of HTTP and HTTPS is that its a fairly bulky header and therefore we often switch to Web Sockets once communication has been "set-up".
    In the exercise below we show how to set up a simple message server which can send messages to a group of people who only can communicate via their browser.

    Module 3 - Containers and Orchestration
    Containers, such as Docker allow users to run images that are independent of operating system. We will use Docker to download an image that already exists in DockerHub and run it in a Docker container on our laptop. This means that anyone can now distribute their code by creating a Docker image. This image can run on any operating system because it will run in a standardized Docker container.
    Now suppose I want to set up several docker containers (lets say 3 containers called A, B and C) on my laptop all running the same code and I want them to all communicate, say on port 3000. If you think about it, this raises a problem. The message header from any container will contain the machine name and the port number. But my machine only has one port 3000 so how does it know which container is its correct destination? The answer is that each container can map its port 3000 to different machine level ports. So container A port 3000 --> 8001, B port 3000 --> 8002, and C port 3000 --> 8003. So a message address to IP X:8002 will go to container B port 3000. This mapping is done in the Dockerfile and its a very powerful mechanism for testing code. (maybe an Exercise on this would be good).

    Module 4 - DevOps
    There are two ways of constructing a software design: One way is to make it so simple that there are obviously no deficiencies, and the other way is to make it so complicated that there are no obvious deficiencies. Quote by C.A.R. Hoare, 1980 ACM Turing Award Lecture
    Today software development has moved from designers throwing their software "over-the-wall" to testers and the testers repeating the process with software operations. Now these roles are disappearing and there are just software engineers who design, test and deploy their software.
    In order to simplify the complexity of the product designers strip it down to the Minimal Viable Product. They don’t put in any feature that is not absolutely necessary.
    Repeated operations are systematized and optimized and decisions are driven by actual data gleened from testing and actual deployment.
